# 11｜测试与调试：让 Skill 稳定复现、稳定修复

关键词：cursor, claude code, skills, skill

Skill 的问题一旦进入“不可复现”，修复成本会指数级上升。所以测试的第一目标不是覆盖率，而是：**把问题变成可复现、可定位、可回归**。

## 1）一套足够用的测试思路

- **固定 3 类输入**：正常输入 / 缺参输入 / 边界输入（大文件、空列表等）
- **固定输出模板**：字段不可缺省，避免“看起来差不多”
- **固定数据源**：例如资源简报只以 `community-resources/` 与 `CHANGELOG.md` 为准

这样做的好处是：每次改 Skill，都能用同一套输入验证输出是否被破坏。

## 2）最常见的 4 类故障

- **不触发**：位置不对、未重启、`description` 没覆盖用户表达
- **触发错**：多个 Skills 触发词过近、边界不清
- **输出飘**：没用模板锁定结构，模型自由发挥
- **脚本失败**：参数没校验、异常没捕获、环境依赖缺失

这些问题几乎都能通过“规范 + 模板 + 结构化输出”解决一大半。

## 3）调试时优先问的三个问题

1. 这次的输入和上次相比差在哪？
2. 数据源变了吗（文件/配置/资源库）？
3. 规则变了吗（指令/模板/脚本版本）？

把问题拆成这三问，定位会非常快。

## 4）判断题（欢迎评论区回复“对/错”）

**判断题：** Skill 的测试重点应优先放在“固定输入与固定输出模板”，而不是一开始就追求很高的自动化覆盖率。（对 / 错？）
