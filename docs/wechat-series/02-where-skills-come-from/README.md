# 02｜Skills 从哪里来：把高频工作流固化成能力（而不是堆技巧）

关键词：cursor, claude code, skills, skill

如果你把 Skills 当成“写作技巧”，很快会陷入两种情况：要么每次都重写一遍，要么写成一个什么都做的“万能 Skill”。更可行的路径是：**从高频工作流里抽象能力**。

## 1）从“重复劳动”里找 Skills 机会

最适合固化成 Skills 的任务，通常满足以下特征：

- **高频**：每周都会做、每个项目都会做
- **有固定产出**：比如日报/周报/清单/结构化报告
- **可拆解步骤**：能写成“先做 A，再做 B，最后输出 C”

本仓库的 `community-resources/` 就是典型：资源会不断增加，但维护动作高度重复——搜索、归档、记录、汇总。

## 2）一个实用方法：先写“边界”，再写“步骤”

很多 Skill 写不稳，是因为一开始就写步骤，却没定边界。建议顺序反过来：

1. **边界**：做什么/不做什么（比如本仓库明确“不收录 Claude Code 安装使用教程”）
2. **输入**：缺什么就问什么（时间段、语言、输出模板）
3. **步骤**：可复现、可检查（数据源在哪里、怎么筛选）
4. **输出**：字段不可缺省（标题、链接、描述、标签…）

你会发现：当“边界”写清楚，后面就很少跑偏。

## 3）从仓库里抄一条“能力链”

仓库已经给出了一条非常清晰的能力链：

- `resource-search`：把“找资源”变成有筛选规则的能力
- `daily-report`：把“今天改了啥”变成可追踪的日报
- `resource-briefing`：把“周期汇总”变成可对外发布的简报

这是把一个持续运营的工作流，拆成三个职责明确的 Skills——比“一个大 Skill 包办一切”更易维护。

## 4）判断题（欢迎评论区回复“对/错”）

**判断题：** 设计 Skills 时，把多个职责拆成多个小 Skill，一般比“一个万能 Skill”更容易长期维护。（对 / 错？）
