# 09｜团队共享：把 Skills 变成可版本化的“团队资产”

关键词：cursor, claude code, skills, skill

个人用 Skills，解决的是效率；团队用 Skills，解决的是一致性与协作成本。关键差别在于：**团队必须能版本化、能评审、能回滚**。

## 1）项目级 Skills：最适合团队

把 Skills 放进项目根目录的 `.claude/skills/`，意味着：

- 新同事拉取仓库就能用同一套能力
- 变更可以走 PR 评审
- 规则可以被历史追踪（谁改了什么、为什么改）

## 2）本仓库的团队协作案例

看 `examples/07-team-shared/`：它展示了“把资源/配置/Skill 一起共享”的项目形态。团队真正需要的不是“某个人会用”，而是“大家都按同一套流程产出”。

## 3）团队最小规范（建议直接写进 README）

推荐至少约束三件事：

- **触发词**：团队约定用词（例如“日报/周报/简报”的口径）
- **输出模板**：固定栏目与字段（便于复制发布/归档）
- **变更策略**：哪些变更算破坏性变更、需要同步公告

没有这些规范，Skills 很容易变成“只有作者自己能用的私人脚本”。

## 4）判断题（欢迎评论区回复“对/错”）

**判断题：** 团队场景下，Skills 最重要的是“功能多”，而不是“输出一致、可审计、可回滚”。（对 / 错？）
