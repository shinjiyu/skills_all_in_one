# 01｜Skills 到底是什么：它解决的是“稳定复用”，不是“更长提示词”

关键词：cursor, claude code, skills, skill

Git 链接：
- 仓库：[shinjiyu/skills_all_in_one](https://github.com/shinjiyu/skills_all_in_one)
- 本文：[docs/wechat-series/01-what-is-skills/README.md](https://github.com/shinjiyu/skills_all_in_one/blob/main/docs/wechat-series/01-what-is-skills/README.md)

很多人第一次接触 Skills，会把它理解成“更高级的提示词”。但真正的差别不在“写得更长”，而在于：**Skills 要解决的是可复用、可维护、可协作的稳定工作流**。

## 1）先把概念说清楚：Skills 是什么

把 Skills 想象成一份“能力说明书”：

- **告诉模型要做什么**：目标是什么、范围是什么
- **告诉模型怎么做**：步骤、规则、异常处理
- **告诉模型怎么交付**：输出格式、字段不可缺省

当你把这些写成结构化内容，并且放进项目里和代码/资源一起维护，它就不再是一次性提示词，而是团队可以复用的“能力资产”。

## 2）Skills 解决的 3 个痛点

- **同一句话每次结果不一样**：你明明说的是“生成周报”，却总有人输出成随笔
- **流程散落在聊天记录里**：换个人、换时间就找不到“当时怎么做的”
- **无法协作和迭代**：想改一条规则，却没有版本记录、没有评审入口

Skills 的价值就是把这些痛点，转成“有结构、有边界、有版本”的可交付能力。

## 3）本仓库给你的最直观例子

在这个仓库里，你能看到两类东西：

1. **示例项目**（`examples/01~08`）：从最小 Skill 到生产级 Skill 的项目形态
2. **项目级 Skills**（`.claude/skills/`）：专门用于维护资源库的 3 个能力（搜索/日报/简报）

它们共同指向一个结论：**Skills 不是“写得漂亮”，而是“能稳定复用”**。

## 4）判断题（欢迎评论区回复“对/错”）

**判断题：** Skills 的核心价值在于“让模型更聪明”，而不是“让工作流更稳定可复用”。（对 / 错？）
